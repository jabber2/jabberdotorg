<DIV><SPAN class=785295316-31032003><FONT face=Arial color=#0000ff 
size=2>Assuming you are running Jabber on your desktop,&nbsp;consider using a 
TCP proxy to forward requests from one port (say 6222) to the Jabber client port

XML stream back and forth.&nbsp; It's a great way to familiarize yourself with 
how the Jabber protocol works, as well as a great debugging 
tool.</FONT></SPAN></DIV>

<DIV><SPAN class=785295316-31032003><FONT face=Arial color=#0000ff size=2>3) The 
UI is client side programming, so whatever the programmer wants they can 
get.&nbsp; Usually the roster display is generated from XML doled out by the 
Jabber server.&nbsp; The Jabber/XMPP spec says additional XML elements can be 
added to the roster (read the spec to see exactly where its OK).&nbsp; An 
important gotcha is that different servers may not honor these additional 
attributes and/or elements, and other clients may not manipulate 
the&nbsp;additional data&nbsp;properly.&nbsp; Quick example is the xdb_sql 
modules running in the Jabber server may not save your new fields into the 
Jabber database, and as such your information would be quickly lost.&nbsp;&nbsp; 
The stock Jabber 1.4.2 server uses files to save XML&nbsp;so you may be 
fine&nbsp;for a while.&nbsp;</FONT></SPAN></DIV>
