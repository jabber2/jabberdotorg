<P><FONT SIZE=2>Maybe I was not verbose enough in my explanation.</FONT>
</P>

<P><FONT SIZE=2>Guaranteed messages is not currently a feature of the Jabber server OR any Jabber clients I know of.&nbsp; It would be a nice _optional_ feature of the Jabber server, but should be optional since it would cause increased traffic that some solutions may not want.</FONT></P>

<P><FONT SIZE=2>Here is a more verbose explanation of what I was proposing using the existing Jabber server version(s) and modifying an existing client or rolling your own:</FONT></P>

<P><FONT SIZE=2>Note:&nbsp; Not showing server in this since it includes not logic for guarenteed</FONT>
</P>

<P><FONT SIZE=2>1) Client A sends message -&gt; Client B </FONT>
<BR><FONT SIZE=2>2) Client B recv message and sends a &quot;receipt message&quot; -&gt; Client A</FONT>
<BR><FONT SIZE=2>3) Client A removes message from &quot;pending receipt&quot; list</FONT>
</P>

<P><FONT SIZE=2>Client A will alert user if a message is on the &quot;pending receipt&quot; list for more than X seconds.</FONT>
</P>

<P><FONT SIZE=2>The &quot;receipt message&quot; could be a standard message with no normal body text but includes an &lt;x&gt; section which includes the message that was received and any other pertinent information.&nbsp; The specialized client would know to process this &lt;x&gt; section to update it's &quot;pending receipt&quot; list.</FONT></P>

<P><FONT SIZE=2>You still maintain the ability to send message to clients that don't support your specialized &lt;x&gt; section, but you will need some way to know (which I believe there is a way to ask for someone's client type) to not wait for return receipts from those clients.</FONT></P>
