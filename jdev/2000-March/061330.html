* Ryan Eatmon (reatmon at ti.com) [000311 06:12]:
> 
> Does anyone think that having a mechanism in place to tell the server to
> dump your offline messages is a good idea?

Maybe.  To me a more useful feature would be to be able to send a message with
some sort of routing flag, indicating that it shouldn't be stored offline at
all.  It seems that would solve your problem, and also make certain other
porgrams much easier/more useful (basically of the sort that would send you a
jabber message to let you know when some condition had occurred, but you
wouldn't want those queuing up when you're offline [a mailcheck daemon, for
instance]).

> I'm writing a program where several daemons are listening to Jabber
> accounts and doing something when they get a message.  The problem is
> that if the daemon is offline, or the client crashes before getting the
> reply then there is a "bad" message in the offline queue.  A mechanism
> where you can send a message to the server and tell it to clear out your
> offline queue might be useful.
