> okay... what about group non-chat? (a la zephyr or gale for instance)
> 
> why not do:
>     group/(chat|popup)
>     single/(chat|popup)

The default would automatically be "popup", chat would have to be
specified as an alternative.  You can have a group transport that works
just like a mailing list, and messages would by default be popup unless
the transport specifies chat.

> I would suggets for clients to implement this (at their descretion) as merely
> the suggested mode. If the user has no mode set for single then default to
> whatever is sent to them as the type. Similar for group, though a client might
> wish to allow for group by group settings. (Low traffic groups would go well
> as chat-style [little scrolling/loss of messages])

Of course, yes, it would be the client's option, this would only be a
suggestion based on the senders settings.

> Also, type seems rather a generic, and possibly a bad choice for the attribute
> name? Might display or mode be better?

Possibly, I'm still debating this myself :)  I guess I'm of the attitude
that we can try using the type="" and as we are progressing with 0.7/0.8
and it is conflicting or there are more display modes, we'll re-evaluate
where it is best placed.
