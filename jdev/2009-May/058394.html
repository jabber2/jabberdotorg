I have a few comments on http://xmpp.org/extensions/inbox/sift.html

The namespace seems to have retained its roots instead of using sift.

I think when a client registers for presence, that is the time when
the server should do presence probes.

I think adding a default attribute to //register/iq is desireable, and
it would work as such:

  default='deny' would mean that the child elements constitute a
whitelist.  default='allow' would mean they constitute a blacklist.
If we think the blacklist use case is not very compelling, then it may
be better to keep it simple.

Some note should be made that if we register for a subset of iqs, it
becomes the server's rsponsibility to respond to unregistered
namespaces so that we remain compliant with 3920.
