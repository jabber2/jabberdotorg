Zhang Yong wrote:
> Consider the following case:
> 
> If user at example.com receives a subscription request ('subscribe') from
> contact at example.net,
> a roster item should be added to the user's roster with the state 'None
> + Pending In'. But after user at example.com rejects the request, or
> contact at example.net 'unsubscribe' to the previous request, the
> corresponding item state in the user's roster should be changed to
> 'None' according to XMPP-IM, thus resulting in a roster item with state
> 'None'.
> 
> But the user has never added such a roster item!!! Is it reasonable?

Whether or not it is reasonable, it's been that way since 1999. ;-)

We could modify that in rfc3921bis, or the client could simply send a 
roster set with subscription='remove' to totally delete the roster item.
