Quoting David Waite <mass at ufl.edu>:
> I think this works well. timezone is optional but includes an offset if it
> is there. The only thing people
> really care about anyways is that they know the other person's local time..

  Yea, exactly..  Heck, even if they don't supply the timezone text, and just 
the offset, a simple database could be made up.  It'd only have what, 24 
entires?

> I was thinking after all this debate, we are going to have client software
> that shows the date prominently,
> perhaps even larger than the actual text message ;-)

  Kinda, yes, but we would now also have the ability to validate message 
routing very well.  And if we EVER get into a situation where we have hosts 
that are sending spam, and getting around it by making the headers look like 
they are from somewhere else, we can add ban filters by where the message was 
routed thru.  And we can have transports actually perform reverse lookups to 
ensure that the route address specified is indeed the one it's coming from.  We 
can really do alot with this data.

  Best part is, it'll be easy to tag this data onto an XML stream, for etherx 
to use as well.
